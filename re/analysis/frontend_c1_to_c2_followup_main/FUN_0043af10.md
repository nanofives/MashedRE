# FUN_0043af10 — Game-setup lobby renderer

**RVA:** `0x0043af10`
**Body extent:** `0x0043af10..0x0043beef` (~3550 bytes, ~370 lines decomp).
**Origin:** D-11025 deferral from `batch-frontend-followup-s2` (oversize cap).

## Signature

```c
void FUN_0043af10(void);
```

Per-frame renderer for the pre-race game-setup lobby panel. No arguments;
all state is read from global option/game-mode/player-status DATs. Called
once per menu frame from the frontend dispatcher.

## Uncertainties

- U-4306: `DAT_007d3ff8 + 0x20` is an RW device-method vtable slot;
  exact RW3 method-table index 0x20 needs cross-ref with RW3 SDK
  header (Criterion `RpDevice`?) to confirm it's `SetRenderState` vs
  some other state slot.
- U-4307: float-immediate constants `1.4013e-45..9.80909e-45` are
  int32(1..7) reinterpreted as floats by the decompiler (same quirk as
  U-4302 for FUN_0040acd0). Original asm is `mov [esp+0x?], <small_int>`.
  Will need un-cast at C3 reimpl time.
- U-4308: the `extraout_ECX/EDX/EDX_00/EDX_01` decomp pseudo-vars are
  artifacts of Ghidra's calling-convention guess (sV calls returning
  values in registers it didn't expect). Inspect each call-site sig at
  C3 time to determine true return convention of FUN_0042aad0,
  FUN_0042b8b0, FUN_0042b8c0.
