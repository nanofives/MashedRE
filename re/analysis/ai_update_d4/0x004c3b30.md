## Mechanical description

1. If `param_1 == 0.0`, returns 0.0 immediately (zero guard).
2. Reinterprets `param_1` as `int`. Adds `0x800` for rounding.
3. `(bits + 0x800) >> 0xc & 0xfff` — extracts bits [23:12] of the float mantissa+exponent as a 12-bit index (0..4095).
4. Table pointer: `*(int *)(DAT_007d3ffc + DAT_007d3ff8)` — first pointer in the RW dispatch array rooted at `DAT_007d3ff8` (same pattern as FUN_004c3df0 from ai_update_d3). Dereferences to get the base of a 4096-entry `int[]` sqrt lookup table.
5. `table[index] + ((bits + 0x800) >> 1 & 0x3fc00000)` — combines the table correction word with the reconstructed exponent bits to form the sqrt approximation as a float bit-pattern.
6. Returns the approximation as `float10` (x87 extended, but value is within float range).

## Asm-level note (added during C3 promotion 2026-05-09)

Same caveat as `0x004c3ac0` Vec3Magnitude and `0x004c3b90` FastInvSqrt:
Ghidra renders the final step `(float)(table_value + exponent_correction)`
as a numeric int→float cast, but the asm at `0x004c3b79..0x004c3b83` is:

```
ADD EAX, ECX            ; EAX = LUT_root_sqrt[idx] + exp_correction (INT add)
MOV [ESP+4], EAX        ; spill
FLD float ptr [ESP+4]   ; reinterpret BITS as float
RET
```

Verified semantics:

```
result = bit_cast<float>( LUT_root_sqrt[idx] + ((biased >> 1) & 0x3fc00000) )
```

LUT root reads at offset `+0` inside the RW globals (same table as
Vec3Magnitude). `biased >> 1` (no NOT) — sqrt exponent contribution.
The only structural difference from Vec3Magnitude is that this function
operates on a single externally-provided float; Vec3Magnitude computes
`v[0]^2 + v[1]^2 + v[2]^2` first and then runs the identical LUT step
on the result.
