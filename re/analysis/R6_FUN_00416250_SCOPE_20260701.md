# WS-R6-C scope: FUN_00416250 (primary per-vehicle AI control step)

Scoped 2026-07-01 (Mashed_pool2 RO), as the next R6 handling/AI verbatim-port target
after WS-R6-A (FUN_00443300 C3, bit-identical) and WS-R6-B (FUN_00443dc0, C2, x87 bug
fixed — see R6_FUN_00443dc0_SCOPE_20260701.md). Anchored MASHED.exe SHA-256 BDCAE0…

## Shape
`void FUN_00416250(int aiLine /*param_1*/, int vehIdx /*param_2*/, undefined1* ctrlOut /*param_3*/)`
Body `0x00416250..0x00416a2e` (~0x7de = ~2014 bytes). __cdecl 3 stack args.
hooks.csv: `00416250 FUN_00416250 ai C2`.

This is the **primary AI control step**: it runs a **priority targeting chain** to pick a
target, LOS-checks each candidate, decides a behavior **mode 0..10** (`local_48`), stores
the mode to `*(0x0089a52c + vehIdx*0x74)`, then computes the steering angle and writes the
**AI control-byte descriptor** `ctrlOut` — matching the WS-C AI ctrl-byte map
([[project_wsc_ai_port]]): `ctrlOut[0]`=steer-left byte, `[1]`=steer-right byte,
`[4]`=accel byte, `[5]`=brake byte (and FUN_00415220 writes fire to `+3`).

## Behavior-mode decision tree (from decomp)
Priority chain, each hit sets `local_48` = mode and (usually) copies target `local_2c→local_24`:
- mode 1: FUN_00414570 (ahead-in-race) + LOS(FUN_00416060) ok
- mode 2: FUN_00415880 (ram-from-behind) + LOS ok + NOT FUN_00415d00 (wall-ahead)
- mode 3: FUN_00414a70 (closest-vehicle) ==1 + LOS ok  (==2 → early neutral return)
- mode 7: FUN_00414c30 (obstacle/bomb avoid) ==2 + LOS ok
- mode 9: FUN_004150e0 (track-wall lateral) ==1
- mode 10: FUN_00426c00()==0x21 + FUN_00414f00 (powerup-seek)
- mode 5: (only if local_34==6 and FUN_00443080()==0 and local_48==0) FUN_00415020 last-place frustration
- mode 8: powerup activation FUN_00415220 (gated on DAT_0088fc88[v*0x2d] powerup type)
Then steering: `FUN_00415e20(vehIdx, tgt.x, tgt.z)` returns the bearing angle in **ST0
(float10)** → drives left/right steer bytes via threshold bands (`_DAT_005cd09c` etc.),
plus turn-history hysteresis in `DAT_008032d8/dc/e0 + v*0x14` and `DAT_0089a4ec/f0/f4 + v*0x1d`.
Tail: mode-specific overrides (7/5/9/2) and a final gate: if game sub-mode `local_34`
(FUN_0040e350 GetRenderSubMode) ∉ {5,6,9,10,11} → zero accel+brake.

## Port caveats (learned in WS-R6-B — read before authoring)
1. **float10 ST0 returns**: FUN_00415e20 (steering angle) returns float10 in ST0; FUN_00443440
   likely too. Any forwarding fn-ptr to a float10/float/double-returning callee MUST be declared
   returning `float` (NOT void) or the SSE2 .asi leaks the x87 stack → NaN + FPU corruption.
   See [[feedback_x87_st0_float10_return_fnptr]]. Check every callee's Ghidra return type.
2. **float10 precision residual**: like FUN_00443dc0, the inline float10 math (`fVar9/fVar10`,
   the threshold-band comparisons) won't be bit-identical from SSE2 C++. Use `double`
   intermediates to emulate x87 single-rounding; expect a small residual; bit-identity needs
   naked-asm. This is a C3 (behaviorally-faithful) target at best in plain C++.
3. **Big surface**: 13 C2 callees, several themselves complex targeting behaviors. A verbatim
   port that FORWARDS all callees to their live RVAs (they exist as originals) is feasible for a
   self-test A/B, but each C2 callee it depends on should ideally be understood. This is a
   multi-increment effort, not a single leaf.

## Recommended approach (WS-R6-C, next increment)
- Author `Ai/AiControlStep.cpp` reimpl of FUN_00416250, forwarding ALL 23 callees to their live
  RVAs with **correct return types** (audit each in Ghidra first — void vs float10-in-ST0).
- Self-test harness like AiNavHooks: snapshot/restore the written state (ctrlOut[0..5], mode
  slot 0x0089a52c+v*0x74, turn-history DAT_008032d8/dc/e0+v*0x14 and DAT_0089a4ec/f0/f4+v*0x1d),
  run orig vs mine per call under an env gate, compare, SAFE PASSTHROUGH default.
- Install at 0x00416250 (MASHED_HOOK_ONLY), track 0, 4 cars. Target GREEN → C3.
- The mode-decision tree is pure integer control flow (should be bit-exact); the divergence
  risk is concentrated in the steering-angle float10 and the threshold-band comparisons.

## Alternative (if verbatim proves too coupled)
Consider whether a smaller, more self-contained R6 leaf is a better next port (e.g. one of the
individual C2 targeting behaviors 00414570 / 00415880 / 00414a70 as its own C2→C3, which also
feeds this function). Draining a few of FUN_00416250's C2 callee leaves to C3 first would
de-risk the orchestrator port and is itself R6-lane progress.
