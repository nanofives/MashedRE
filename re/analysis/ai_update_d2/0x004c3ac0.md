## Mechanical description

1 arg: `(param_1=float* vec3)`. Returns float10.

- Computes `sq = param_1[2]^2 + param_1[1]^2 + param_1[0]^2` (sum of squares) (`0x004c3acc`).
- If `sq == 0` → return 0.0 via cast.
- Two-level lookup table fast-sqrt:
  - Level 1: `*(int *)(*(int *)(DAT_007d3ffc + DAT_007d3ff8) + ((sq + 0x800) >> 12 & 0xfff) * 4)` → coarse table pointer.
  - Level 2: `*(float *)(level1 + ((sq + 0x800) >> 1 & 0x3fc00000))` → fine lookup.
- Returns approximate `sqrt(sq)` as float10.

## Summary

Fast 3-vector magnitude: computes Euclidean length of a float3 using a two-level floating-point sqrt lookup table at DAT_007d3ff8/007d3ffc. Zero-guard returns 0.0. Used throughout AI for distance checks.

## Asm-level note (added during C4 verification 2026-05-09)

Ghidra's pseudocode renders the final lookup as `*(float *)(level1 + ((sq + 0x800) >> 1 & 0x3fc00000))`. That reads as a memory dereference but the actual asm at `0x004c3b17..0x004c3b2f` is:

```
MOV EDX, [EAX + ECX*4]   ; EDX = LUT_root[idx]   -- a 32-bit value, NOT a pointer
SHR EAX, 1; AND EAX, 0x3fc00000   ; EAX = exponent_bits
ADD EDX, EAX             ; EDX = mantissa_bits + exponent_bits  (INT add)
MOV [ESP+4], EDX         ; spill
FLD float ptr [ESP+4]    ; <-- reinterpret bits AS float, NOT *(float*)EDX
```

So the verified semantics are:

```
result = bit_cast<float>( LUT_root[idx] + level2_off )
```

— the classic RW3 fast-sqrt where `LUT_root[idx]` holds precomputed mantissa-side bits keyed on the high 12 bits of `(sq_bits + 0x800)`, and `(sq_bits >> 1) & 0x3fc00000` contributes the exponent-side bits. The two halves combined form a valid IEEE 754 float bit pattern that approximates `sqrt(sq)`.

A literal C translation of Ghidra's pseudocode (`*(float*)(level1 + level2_off)`) crashes immediately because `level2_off` can be ~0.5 GB and is meaningless as a byte offset. The bit-cast interpretation was verified by Frida A/B diff, 18/18 test vectors bit-identical against the live LUT in MASHED.exe (`log/diff_vec3_magnitude.csv`).
