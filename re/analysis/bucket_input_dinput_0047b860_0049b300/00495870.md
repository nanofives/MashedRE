## Mechanical description
Per-joypad poll + cooked-axis computation for one device. `this` arrives in ESI
(decompiler shows `unaff_ESI`; calling convention unrecovered). No explicit
parameters, no return value. 1543 bytes. Stack cookie present
(`local_4 = DAT_00616038 ^ unaff_retaddr`). Field indices below are dword
indices into the `this` record unless noted; the per-joypad record stride is
0x112 dwords (cross-ref 00495e80 / 004957a0).

1. `piVar2 = (int *)*unaff_ESI;` — device interface pointer = first dword of the
   record.
2. **Poll**: `(**(code**)(*piVar2 + 0x64))(piVar2)`. If result `< 0`, enter a
   **re-acquire** loop: repeatedly call `(**(code**)(*piVar2 + 0x1c))(piVar2)`
   while the result equals `-0x7ff8ffe2` (= 0x8007001E).
3. **GetDeviceState**: `(**(code**)(*piVar2 + 0x24))(piVar2, 0x50, auStack_58)`
   — fills an 0x50-byte state block. On `< 0`, `return` immediately.
4. **History ring shift (block A)**: 15-iteration loop sliding paired dword lanes
   downward across `this+0xd0` and `this+0x111` regions (each iteration copies
   `[-1]→[0]` / `[-2]→[0]` / `[0x1e]→[0x20]` etc.).
5. Zeroes dword fields `this[0xf2]`, `this[0xb3]`, `this[0xb2]`, `this[0xd3]`,
   `this[0xd2]`.
6. `FUN_004b6480(this+0x102, 0x20)` — zero-fills 0x20 bytes at `this+0x102`.
7. **Button bitmap**: an 8-step unrolled loop (`iVar14` 1→0x11 step 8) tests the
   sign bit (`< '\0'`) of bytes in `acStack_35[...]` (button bytes of the state
   block) and, for each pressed button, OR-sets the corresponding bit in the
   bitmap at `this+0x408` (`bit = 1 << (idx & 7)`, byte = `idx >> 3`).
8. **Axis pre-scale**: `this[0xb2] = (float)(iStack_64 >> 0xc) * 0x3e088889 - 1.0`
   and `this[0xb3] = 1.0 - (float)(iStack_60 >> 0xc) * 0x3e088889` (signed
   arithmetic shift right by 12; scale 0x3e088889 ≈ 0.133333; bias 1.0 from
   0x005cc320).
9. **POV/hat decode** from `iStack_44`: if `(short)iStack_44 == -1` skip (no hat).
   Otherwise exact-match the centidegree value and write a literal (x,y) float
   pair into `this[0xb2]`/`this[0xb3]`:
   - 0 → (0.0, 1.0); 9000 → (1.0, 0.0); 18000 → (0.0, -1.0); 27000 → (-1.0, 0.0)
   - 0x1194 (4500) → (1.0, 1.0); 0x34bc (13500) → (1.0, -1.0)
   - 0x57e4 (22500) → (-1.0, -1.0); 0x7b0c (31500) → (-1.0, 1.0)
   (1.0 = 0x3f800000, -1.0 = 0xbf800000; the 4500/22500 cases set both lanes to
   the same value via `uVar9`.)
10. **Deadzone gate**: for each lane, `fVar = 0.0` (from 0x005d757c) unless
    `_DAT_00772fa8 <= ABS(lane)` in which case `fVar = lane`. Stores back.
11. **Magnitude clamp**: `mag2 = x*x + y*y`; if `1.0 < mag2`,
    `s = 1.0 / SQRT(mag2)` and multiply both lanes by `s` (clamp to unit circle).
12. **History ring shift (block B)**: 7-iteration loop sliding paired float lanes
    across `this+0xa0` / `this+0x0e`/`+0x10` regions.
13. Stores current cooked pair: `this[0x92] = x`, `this[0x93] = y`.
14. **Moving average**: zeroes then re-accumulates `this[0xb2]`/`this[0xb3]` by
    summing the 8 historic float pairs at `this[0x94..0xa1]` (+ `this[0xa0/0xa1]`)
    then multiplies both by `0x3e000000` (= 0.125 = 1/8). Returns (void).

Mechanically this is: poll DInput device → re-acquire on input-lost → read state →
shift history rings → build a button bitmap → derive a unit (x,y) vector from
axes or the POV hat → deadzone + unit-clamp → 8-sample smoothing.
