# R6 handling v2 — what is PORTED-shape vs SCAFFOLD (2026-06-10)

Source: existing C1/C2 notes, primarily `re/analysis/vehicle_update_d2/00470670.md`
(VehicleControlUpdate, 0x00470670). The original's car-state struct (offsets from
the vehicle object base, all cited in that note):

| offset | meaning | adopted in standalone model |
|---|---|---|
| +0x9b0..0x9b8 | velocity vec3 | YES — `car_vel_[3]` replaces the scalar speed |
| +0x9d4..0x9dc | forward vec3 | YES — derived from yaw |
| +0x190 | per-vehicle max-speed constant | shape adopted (tuning scaffold) |
| +0x1a8 / +0x26c | throttle force accumulators (force along forward) | YES — throttle applies force along forward |
| +0xb0c | `(dt − |dot(fwd,vel)|/speed)·speed` = NON-forward (lateral/slide) speed measure | YES — grip model bleeds the lateral velocity component |
| +0xb24/+0xb28 | boost timers | not yet (no boost input) |

ADOPTED (struct + update shape, from the decomp): velocity-vector state; throttle
as a force along forward; brake/reverse as the negated pipeline; slide measured as
the velocity component orthogonal to forward; per-tick integration with delta-time
scaling.

SCAFFOLD (tuning, not yet harvested/diffed): the actual force constants
(_DAT_005ceaa8 throttle scale, _DAT_005cc950 boost, surface grip tables), the
wheel-force integrator (FUN_0046ddb0) and slope interaction via RW-Physics.
The standalone uses: grip = exponential bleed of lateral velocity; slope = gravity
projected onto the ground plane (normal from the collision triangle); drag ∝ v.
Telemetry diff vs the original is DEFERRED to the physics-gate session (needs the
in-race scenario harness driving matched inputs).
