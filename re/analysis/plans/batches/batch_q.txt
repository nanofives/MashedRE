================================================================================
BATCH Q — promote-c2 wave 3 (standalone-exe critical path: boot + frame-dispatch
            + frontend-draw callees)
================================================================================

SIXTEEN parallel sessions. Pool slots pre-assigned Mashed_pool0..Mashed_pool15
(one per session, no race on `acquire`).

PREREQUISITE: any in-flight ghidra-sweep must complete before this batch
starts. Sessions cross-reference earlier batch_m/n/p plates that must be
in master.

GOAL — RAISE C2 FLOOR ON THE STANDALONE-EXE CRITICAL PATH

The standalone target (mashed_re.exe → drawn main menu without MASHED.exe)
needs the boot chain, frame loop, and frontend-draw chain all at C3. Each
C3 promotion requires its callees at C2+ (caller-gate per re/CONFIDENCE.md).
Today most of those callees are still C1, which is why recent C3 attempts
(c3-batch-d, parts of e) burned cycles on caller-gate refusals.

This batch promotes 16 × 7 = 112 RVAs from C1 to C2 across three clusters:
  A.  WinMain depth-2 chain (boot subsystem)            sessions 97..102
  B.  Frontend-draw FUN_0043c5b0 direct callees         sessions 103..108
  C.  Frame-dispatch FUN_00492e90 callees +
        camera-matrix cluster                           sessions 109..112

A C1→C2 promotion is mechanical: read the existing C1 plate + decomp,
assemble a `## Why C2` block citing constants/offsets at RVA, list
explicit [UNCERTAIN] tags with U-IDs, file new stubs for any unmapped
callees, and flip hooks.csv `confidence` C1→C2. Same per-session shape
as batch_n.

All sessions follow re/SESSION_RULES.md § "Parallel-fanout scribe-queue
pattern". After all 16 finish committing, run `ghidra-sweep`.
