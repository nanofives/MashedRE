WHY THIS BATCH HAS HIGHER CANDIDATE COUNTS THAN c3_batch_i
----------------------------------------------------------
This batch applies the FOUR filter-implementation fixes flagged by the
c3_batch_i orchestrator post-mortem:

1. Inline [UNCERTAIN] regex fix — c3_batch_i used `\[UNCERTAIN\]` which
   misses `[UNCERTAIN U-xxxx]` (mandatory `]` constraint). Filter j uses
   `\[UNCERTAIN[ \]]` which catches both bare and catalogued forms.
2. Catalogued-U-ID exemption — per re/CONFIDENCE.md, inline `[UNCERTAIN
   U-xxxx]` markers whose U-IDs are catalogued in UNCERTAINTIES.md should
   NOT block C3. Filter j cross-references UNCERTAINTIES.md (1,121 U-IDs)
   and only refuses bare/uncatalogued markers.
3. Body-grep callee extraction scope — c3_batch_i's filter grepped RVAs
   anywhere in the note body, catching false-positives from "Callers",
   "Stubs encountered", and "Resolves U-xxx from <prior_note>" sections.
   Filter j restricts grep to `## Decompilation` / `## Callees` /
   `## Mechanical description` sections only.
4. "Resolves [UNCERTAIN U-xxx] from <prior_note>" prose pattern — this
   form (common in promote_c2_vehicle_b notes) was making the body-grep
   filter mistakenly count cited resolved-by-this-note RVAs. Filter j
   explicitly skips lines matching `Resolve[ds]? \[UNCERTAIN`.
