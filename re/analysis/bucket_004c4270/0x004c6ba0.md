## Mechanical description

- Signature `int* f(name1, name2)`. Entry-point for texture load with
  full mipmap chain.
- Step 1: copies `name1`/`name2` into 256-byte stack buffers (with
  error-on-overflow as 0x004c6750).
- Step 2: computes feature flags `uVar8` based on
  `*(DAT_007d4054+0x1c)` (has-pixel-shader?) and
  `*(DAT_007d4054+0x20)` (has-compressed-tex?). Final value is one of
  `0x0004 | 0x8004 | 0x9004` — RW texture format hint bitset.
- Step 3: optional rebuild-filename callback at
  `DAT_007d4054 + 0x30`.
- Step 4: calls **`FUN_004c6750`** (master single-image loader) to
  load the base mip into `iVar3`.
- Step 5: creates the platform raster via `FUN_004c77c0` (RwRasterCreate
  — verified at hooks.csv 0x004c77c0 as `texture_loader_d3`).
- Step 6: if the raster reports "user-loaded mipmaps" (bit 0x80 in
  upper byte of flags, line 109), iterates mip levels:
  - For each level i > 0, computes per-mip filename via the rebuild
    callback (using i as the mip suffix index).
  - Calls FUN_004c6750 for the per-mip image.
  - Stores into `aiStack_640[i]`.
- Step 7: if the raster has mip-compression bit (`& 0x40`), calls
  `FUN_004c6580` (palette unification) for the full chain.
- Step 8: for each mip, copies pixel data into the raster via
  `FUN_004d5310` (RwRasterLockMipLevel + memcpy). Destroys the
  source image via `FUN_004cdd00`.
- Step 9: allocates a new RwTexture object via
  `(DAT_007d3ff8 + 0x118)(*(DAT_007d4054 + 8), 0x30006)` — plugin
  malloc of size 0x30006 (the RwTexture allocation tag).
- Step 10: initialize texture fields: +0x14 (raster), +0x51
  (filter+addressing flags = 0x11), +0x4 (name string slot, 0x20 max),
  +0xc (mask-name string slot, 0x20 max), +0x50 (refcount = 1).
- Step 11: adds the new texture to the plugin registry via
  `FUN_004d8000(&DAT_00618138, texture)`.
- Mechanical shape: the **complete** RwTextureRead pipeline, mip-aware
  and platform-aware. This is the public RwTextureRead entry-point.
