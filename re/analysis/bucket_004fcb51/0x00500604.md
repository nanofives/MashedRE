Recognises C-style leading-zero octal literals (`0[0-7]*`) — but note the prefix is `'0'` alone (no `'x'`). Detects 32-bit overflow by watching the top 3 bits before each `* 8`. Emits warning 0x3eb `"octal value truncated to 32bits"` on overflow.

### Note on the prefix

The first character is required to be `'0'`. The loop starts by advancing _past_ this `0`, then accepting `0..7` digits. So `"0"` alone returns 1 (consumed the `'0'`, no further digits, no overflow). `"077"` returns 3. `"08"` returns 1 (only the `'0'`; the `'8'` is left for the caller).

This matches the C/HLSL grammar exactly: a bare `0` is octal zero; `0NNN` with `N <= 7` is octal-N; `08NN` is invalid as octal but parses as `0` plus the rest.

### Dispatch position in FUN_00500adc

`FUN_00500604` is tried _after_ `FUN_0050055f` (hex) in the integer-literal chain. Because the hex check requires `0x`, the octal-on-`0` overlap is safe — if the leading-`0` is followed by `x`, the hex lexer has already consumed it. If by `0..7`, the octal lexer consumes it. If by `8..9`, only the bare `'0'` is consumed and the caller falls through to `FUN_00500680` (decimal) — which will then re-scan starting at this `'0'` and produce a decimal value. The redundant decode is the expected MSVC C grammar shape.
