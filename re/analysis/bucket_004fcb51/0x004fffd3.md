## Mechanical description

**Integer-literal suffix consumer** (`__thiscall(this, source_ptr, out_typecode_or_null)`).

Scans forward greedily over `u`/`U` and `l`/`L` byte suffixes (in any case-insensitive order) and writes one of two type-codes:

| Suffix(es) observed | Out code | Inferred meaning |
|---|---|---|
| `u` (and optionally `l`) | 4 | unsigned (matches FUN_005002ec case 4 "integer '%uul'") |
| `l` only | 3 | long (matches FUN_005002ec case 3 "integer '%dl'") |
| no suffix consumed | (untouched; caller defaults to 2 — "integer '%u'") |  |

`out_typecode` is only written when at least one suffix character was consumed. Returns the number of source bytes consumed.

### Flow

`bVar1` (saw-U) and `bVar2` (saw-L) start false. While cursor < end:

1. If `bVar1` already true: only `'l'`/`'L'` next is accepted (sets `bVar2`); otherwise break.
2. If `bVar1` false: try `_tolower == 'u'` (sets `bVar1`); if not, fall through to the `bVar1 == true` branch (which immediately tries `'l'`).
3. On success, advance cursor.
4. Loop until break or end.

After loop: if `bVar1` → `*param_3 = 4` (unsigned); else if `bVar2` → `*param_3 = 3` (long); else leave `*param_3` untouched. Return `cursor - param_2`.

Caller `FUN_00500adc` calls this with `iVar2 = FUN_004ffc86 == 0` having succeeded as an integer literal in FUN_0050055f/FUN_00500604/FUN_00500680 (hex/octal/decimal). The initial `*param_3 = 2` write in FUN_00500adc establishes the default "plain integer" type-code, which this routine bumps to 3 (`l`) or 4 (`ul`/`u`) on suffix match.

### Note on accepted shapes

The loop pattern is `(u | u? l)`: a single `u` (optionally followed by `l`) gives code 4; a single `l` (without `u`) gives code 3. **Doubled `ll` or `LL`** is _not_ specifically recognised — the second `l` would fall through with `bVar2` still true (already-saw-L is not retested, the `bVar1 == true` branch only checks the `'l'` once). The grammar is the legacy single-`L` integer suffix.
