## Mechanical description

**Float-literal suffix consumer** (`__thiscall(this, source_ptr, out_typecode_or_null)`).

Looks at the byte at `param_2`. If it is `'f'`/`'F'` or `'h'`/`'H'`, consumes one byte and writes a type-code:

| Suffix | Out code | Inferred meaning |
|---|---|---|
| (none / not matched) | 5 | default float kind |
| `'h'` / `'H'` | 6 | half / horizontal float |
| `'f'` / `'F'` | 7 | (typical C-like `f` float suffix) |

Returns the number of bytes consumed (0 if cursor at end, 1 if suffix consumed, 0 if suffix not matched and `out_typecode` left at the default 5 — actually returns `pcVar2 - param_2` which is 0 when `pcVar2 = param_2`).

### Flow

1. End-pointer guard `*(this+4) <= param_2` → return 0.
2. `_tolower(*param_2)`:
   - `== 'f'` (0x66) → out=7, advance 1.
   - `== 'h'` (0x68) → out=6, advance 1.
   - else → out=5, no advance.
3. If `param_3 != NULL`: write `*param_3 = out`.
4. Return `pcVar2 - param_2` (0 or 1).

Caller `FUN_00500adc` calls this with `iVar2 = FUN_004ffc86(...)` having just succeeded as a float literal — so this is the "now read the trailing `f`/`h` suffix" pass for HLSL-style float kinds. The base type-code 5 matches the `case 5` ("float '%g'") error-printer in FUN_005002ec; codes 6 ("float '%gh'") and 7 ("float '%gf'") match the same printer's float-with-suffix branches.
