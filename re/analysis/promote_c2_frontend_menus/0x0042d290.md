## Addendum 2026-05-21 — format-string semantics corrected via runtime evidence

The original "Globals accessed" table claimed DAT_005cd798 was `"%02d"` and DAT_005cd7a0 was a "pad value." Frida A/B harness output (log/diff_menus_lap_time_fmt.csv, GREEN 10/10 after the C++ reimpl bug fix at MenuMixed.cpp:404-419) shows:

- `DAT_005cd794` → format string `"%d"` (>= 10 path; address-is-string)
- `DAT_005cd798` → format string `"%s%d"` (< 10 path; two specifiers)
- `DAT_005cd7a0` → C-string `"0"` (pad-prefix passed as %s arg)

Ghidra's high-level decomp shows `FUN_004a2b60(param_3, &DAT_005cd798, &DAT_005cd7a0)` for the `< 10` path, but FUN_004a2b60 is variadic (vsprintf-style) and Ghidra's prototype elides the 4th variadic arg. The actual call shape is:

```c
// < 10 path: 4 args (variadic):
FUN_004a2b60(param_3, "%s%d", "0", param_1);  // outputs e.g. "07" for param_1=7
// >= 10 path: 3 args:
FUN_004a2b60(param_3, "%d", param_1);         // outputs e.g. "42" for param_1=42
```

Same dispatch for the second buffer with `iVar1` (FUN_004a2c48 return). Evidence: bit-identical Orig vs reimpl output across {0, 1, 5, 9, 10, 42, 99, 100, 1234, -1} after fixing the two reimpl bugs (format-string indirection and variadic arg count).
