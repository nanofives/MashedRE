### float10 caveat (why C++ won't be bit-identical, only behaviorally faithful)
The ray-march marching param (fVar11) and the (1.0/len) factor are kept in 80-bit ST
across the loop; sample coords store to float32 then round-to-nearest→__ftol→tile. A C
`float` reimpl rounds the marching param each iter → sample positions differ in low
bits → occasionally a different tile at a boundary → occasional wall-detect / output
divergence. EXPECTED: mostly-GREEN with rare boundary divergences (gameplay-inert).
This is C3 (behaviorally faithful); bit-identity would need naked-asm (994 instr).

## ROOT CAUSE + FIX (2026-07-01, debug session) — divergence was H1 (a real bug), residual is H2

The measurement-#1 divergence was NOT purely the float10 path. Bisection found a genuine
**correctness bug** (H1) sitting on top of a smaller float10 residual (H2). Ghidra facts
(Mashed_pool2 RO): **`FUN_004c3c60` (Vec2Normalize) and `FUN_004c3bf0` (Vec2Length) both
RETURN `float10` in ST0** (verified signatures). `FUN_00443300` (Spline) and `FUN_00416230`
(Arm) return `void` (no ST0).

- **BUG (H1):** AiNavHooks declared the Vec2Norm fn-ptr as `void (float*,float*)`. On x86 a
  float/double/float10 return lives in ST0 and the caller MUST pop it (fstp) even when unused.
  A `void` decl in the SSE2-built .asi never emits the pop → ST0 leaks 1 slot per Vec2Norm
  call. The forward walk calls Vec2Norm **2×/iteration**, overflowing the 8-deep x87 stack
  after ~2 iters → every later x87 result is the indefinite NaN `0xffc00000`. Effect:
  walk candidates 2..15 had **NaN x + NaN score** (z was still valid — proving the walk
  itself was correct), so the argmax could only ever pick candidate 0/1 (never the far
  lookahead) → systematic wrong target + arm-flag flips. The unbalanced stack on RETURN
  also corrupted the caller's FPU → **the whole AI/physics froze** (cars didn't drive).
  Vec2Len was already declared `float` (its return is used), so it popped correctly; that
  is why the seed/subdivision (Spline-only) never overflowed and the cache always matched.
- **FIX:** declare the Vec2Norm fn-ptr as returning `float` (value return) so the compiler
  pops ST0 after each call. Also promoted the decision-driving dist^2 / dot / subdivision /
  ray-march arithmetic to **`double` intermediates** (single round to float32 at the store)
  to emulate x87's single-rounding vs SSE2's per-op double-rounding.

### Residual is H2 (float10) — bit-identity needs the naked-asm body
The remaining flips are the predicted float10 issue: the dot/dist^2 run in `double` (~2^-53)
not true x87 80-bit (2^-64). It is **all-or-nothing**: the whole chain
`subdivision-winner-t → forward-walk-t → Spline → Vec2Norm → dot → argmax` feeds the
selection, so a `double`-vs-x87 diff anywhere perturbs the near-tied argmax. Partial inline
x87 for just the dot cannot close it. Bit-identity therefore requires the **full naked-asm
x87 verbatim body** (994 instr), exactly as scoped in "Port plan" above. Deferred.
