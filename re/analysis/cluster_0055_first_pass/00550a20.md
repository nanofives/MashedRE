## Mechanical description

`gets`-equivalent on an open RtFS file handle (`param_3` = file handle;
`param_1` = output buffer; `param_2` = max bytes).

1. Calls `(*(*(param_3+0x38)+0x30))(param_3, param_1, param_2-1)` —
   read-impl from the FS vtable (file handle's FS pointer at +0x38, then
   vtable+0x30). Returns bytes read (iVar1).
2. If 0 bytes read, returns 0.
3. Scans the buffer for the first newline:
   - If `\n`: terminate buffer at next byte; if file handle has +0x18
     non-zero, set +0x28=4, +0x20=3 (pending op = "rewind to line start");
     call `(*(*(param_3+0x38)+0x38))(param_3, (idx+1)-bytes_read, 2)` —
     seek-impl with `whence=2` (SEEK_END-relative back-up) to rewind
     buffered bytes past the newline. Returns param_1.
   - If `\r` followed by `\n` (and idx < limit): shift the rest of the
     buffer down by 1 byte (dword-copy then tail-copy), decrement iVar1.
     Used to collapse `\r\n` -> `\n` (text-mode line-ending normalization).
4. If no newline found: NUL-terminate the buffer at iVar1 and return param_1.

- `2` at 0x00550aa1 — seek whence (SEEK_END-style).

## Subsystem hypothesis

RtFSManager: `RtFile_gets`-style line-reader. Per-FS implementations supply
read and seek; this routine handles text-mode `\r\n`-> `\n` normalization
and partial-line pushback.

## Uncertainties

- [UNCERTAIN] whence semantics — `2` is conventionally SEEK_END, but the
  pushback offset is computed as `(idx+1)-bytes_read` which is negative
  and only makes sense as SEEK_CUR-relative. The FS-impl may interpret
  `2` differently from libc. Filed `U-0055-006`.
